## medium: Project cannot install on supported Node 24 because @types/node is pinned to Node 20 types

id: fnd_sig-feat-config-7528cb5b98-c3591_28ae3b07d0
category: build-release
confidence: medium
triage: risk
status: open
feature: Project config package.json (feat_config_7528cb5b98)

evidence:
- package.json:7
- package.json:58

The package declares Node 24 as a supported runtime, but the TypeScript ambient Node API package is constrained to Node 20. That can make typechecking diverge from the advertised runtime surface: Node 22/24 globals and API signatures may be missing or typed differently, while APIs removed or changed after Node 20 may still appear valid. This is a release/config hazard because contributors and CI can get misleading type results for a runtime the package explicitly supports.

recommendation:
Align @types/node with the supported runtime range, for example by using a Node 22-compatible type package if CI standardizes on Node 22, or by documenting and testing separate typecheck jobs for each supported major.

test analysis:
No typecheck command or CI matrix is included in the reviewed evidence, and the test script only runs selected test files rather than validating runtime type declarations across supported Node majors.

suggested regression test:
Add a typecheck script and run it under the supported Node major used for builds, or add CI jobs for Node 22 and Node 24 if both are intentionally supported.

minimum fix scope:
Update package.json devDependency/runtime support policy and add or document the matching typecheck coverage.

## medium: Test script only discovers two hard-coded globs, so new tests outside those folders are silently skipped

id: fnd_sig-feat-config-7528cb5b98-da0c4_2f931e9575
category: test-gap
confidence: high
triage: test-gap
status: open
feature: Project config package.json (feat_config_7528cb5b98)

evidence:
- package.json:16 (scripts.test)

The test command enumerates only top-level tests in src/lib and src/components/analysis. Any tests added under other common project areas, nested folders, app routes, hooks, utilities, or other components will not run when contributors or CI invoke npm run test. This creates a concrete regression risk because passing tests can falsely indicate full project test coverage while excluding newly added test files.

recommendation:
Use a broader recursive test pattern that matches all project test files, such as src/**/*.test.ts and src/**/*.test.tsx, with any necessary exclusions for generated or fixture files.

test analysis:
The gap is in the test discovery command itself; tests outside the two hard-coded globs would not execute and therefore cannot reveal their own omission.

suggested regression test:
Add a trivial test file under another expected location, such as src/components/example.test.tsx or src/app/example.test.ts, and verify npm run test executes it before accepting the discovery pattern change.

minimum fix scope:
Change package.json scripts.test to a recursive project-wide test discovery pattern.
