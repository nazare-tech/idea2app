Thesis

MVP Summary

01 / 11

Wear It Now AI Mirror is a Chrome extension that replaces product images on clothing sites with AI-generated photos of the user wearing those clothes. It targets fashion influencers (10k–500k followers) who need to evaluate and showcase garments across retailers without buying them. The core problem is style confidence: flat product photos don't show how clothes look on a specific body.

This MVP validates one critical assumption before anything else: can AI output quality reach a threshold influencers would trust enough to download, share, or use in content? If the output looks cheap or uncanny, the product collapses regardless of how well the extension is built.

Validation

The Bet

02 / 11

Goal

Validate whether AI try-on quality is high enough that influencers would download, share, or use the result — and whether they'll pay credits to generate more.

Definition of Done

A user can upload their photo, add a product image, generate a try-on, and download it — without help — and the riskiest assumption has been testably exposed.

Riskiest Product Assumption

The AI output is realistic enough that influencers trust it for content or affiliate showcasing. If outputs look uncanny, users won't pay for credits regardless of UX.

Riskiest Technical Assumption

The image-to-image model produces consistent, photorealistic garment-on-person results within an acceptable window (<30s) using only a single reference photo and a product image.

Audience

Target User & Problem

03 / 11

Primary user

Mid-tier fashion influencers (10k–500k followers) who post outfit content on Instagram or TikTok and earn income through affiliate links.

The problem

They can't tell how a garment looks on their body from flat photos — so they buy speculatively (wasted spend) or skip items (lost affiliate revenue).

Current workaround

Buy, photograph, return if it doesn't work — or skip entirely. Generic retailer try-on tools don't cover the brands they actually shop.

Desired outcome

See a realistic, shareable image of themselves in any garment from any retailer — fast enough to use in their content workflow without buying first.

Scope Decisions

Key Assumptions & Scope

05 / 11
High Confidence
Fashion influencers are the primary user
Every scope decision prioritizes their workflow.
Validation
A full extension is not the first step
AI output quality is the riskiest assumption. v1 is a standalone web page where users upload a photo + product image and the AI generates a try-on — testing quality before any extension infrastructure.
Assumption
One avatar photo per user is enough
The PRD confirms no multiple saved profiles for the MVP.
Scope
Only four capabilities ship
The PRD lists role-based access, generation history, download tracking, and share analytics. The MVP includes only avatar upload, single try-on generation, credit purchase, and result download. Everything else is deferred.
Scope
The extension is deferred to Phase 2
It's the target delivery format, but it adds 2–4 weeks and Chrome Web Store review risk before the core assumption is validated.
Compliance
Uploaded photos are personal data
Don't store raw uploads beyond the session unless the user opts in; store only the output image, show a clear data-use disclosure at upload, and never train on uploads without consent. Defer persistent avatar storage until privacy review.
Conflict Resolved
Overriding the “no web app” non-goal
The PRD lists it as a non-goal, but a minimal web prototype is the fastest way to test output quality without Chrome Web Store delays. The extension remains the target — this is a sequencing decision, not a direction change.

Discipline

AI Build Guardrails

08 / 11

Process

  • Build one chunk at a time. Do not skip ahead.
  • Do not add features outside MVP scope.
  • Do not build the extension, generation history, share-to-social, or admin dashboard.
  • Do not introduce new libraries unless necessary. Ask first.
  • Before changing anything, inspect the codebase and summarize the current architecture.
  • Use mock data before connecting real backend logic.
  • Keep files under 200 lines where practical.
  • Reuse existing components. Avoid duplicate logic.

Quality & safety

  • Add loading, error, empty, and success states for every core flow.
  • Prioritize a working core flow over visual polish.
  • All AI API calls go through server routes. Never expose Replicate or Stripe keys client-side.
  • Do not refactor unrelated files.
  • After each chunk: list files changed and explain how to test locally.
  • Never store raw avatar photos without explicit consent and a visible data-use disclosure.
  • Do not train on uploaded photos — requires separate legal review and explicit opt-in.
  • Do not add admin functionality in this phase.

Validation Plan

09 / 11

First test audience

10–15 fashion influencers with 10k–200k followers who actively post outfit content and use affiliate links on Instagram or TikTok.

How to find them

DM mid-tier accounts found via “OOTD” search, or post in fashion-influencer and creator-economy communities. Offer 20 free credits for 15 minutes of feedback.

Success signals

Strong

Generates an image, downloads it, and shares it or says they'd use it in content; purchases credits unprompted or asks how to get more.

Weak

Completes the flow but says output quality isn't good enough to share publicly; engages only when prompted.

Pivot

Understands the product but says the problem isn't painful enough to pay for, or prefers buying the item anyway.

Onboarding

Key feedback questions

  • When you first saw the generated image, what was your honest reaction?
  • Is this output quality good enough that you'd post it or share it with your audience?
  • Where did you feel confused or stuck during the process?
  • What would need to change about the output for you to use this regularly?
  • Would you pay for credits to keep using this, and what price feels right?
  • What clothing sites do you shop most — and would you want this to work there automatically?

Cut List

10 / 11
Replicate latency is too high or outputs are poor quality.
Test 3–5 try-on models on Replicate before committing; swap models without changing the UI.
Stripe Checkout integration takes too long.
Send Payment Links manually for the first 20 purchases; add embedded checkout in Phase 2.
Avatar storage & privacy handling becomes complex.
Skip persistent storage in v1; require re-upload each session until privacy review is complete.
Product URL parsing breaks on certain retailer sites.
Accept file upload only in v1; URL parsing is a Phase 2 problem.
Credit system logic becomes complex.
Store credits as a single integer in Supabase; no ledger, no auto-reload, no tiers.

first-version-prompt.txt 
You are helping me build the MVP for Wear It Now AI Mirror.

# Product type
Primary: B2C Consumer, Secondary: AI-first Product

# Recommended first version
AI prompt/output prototype — test generation quality before building any UI

# MVP goal
Validate whether AI-generated try-on output quality is high enough that fashion
influencers would download, share, or use the result in content, and whether they
will pay credits to generate more.

# Definition of done
The MVP is complete when a user can upload their photo, provide a product image,
generate a try-on result, and download it without help — and the riskiest assumption
(AI output quality) has been testably exposed.

# Target user
Mid-tier fashion influencers (10k–500k followers) who post outfit content on
Instagram or TikTok and earn income through affiliate links.

# Core user flow
1. User lands on the web app
2. User signs up with Google OAuth or magic link
3. User uploads one photo of themselves (avatar)
4. User pastes a product image URL or uploads a product photo
5. User clicks "Try It On" — one credit is consumed
6. AI generates a photo-realistic image of the user wearing the garment (under 30 seconds)
7. User views the result, downloads it, or purchases more credits

# Tech stack
- Frontend: Next.js + Tailwind + shadcn/ui
- Backend: Next.js API routes (server-side only for all AI and payment calls)
- Database: Supabase (auth, user records, credit balances, output image URLs)
- Auth: Supabase Auth — Google OAuth + magic link
- AI/API: Replicate (IDM-VTON or equivalent try-on model)
- File Storage: Supabase Storage
- Deployment: Vercel

# Build only this first chunk
Set up a standalone test harness (no UI required) that calls the Replicate try-on
API with a hardcoded sample user photo and a hardcoded product image. Log the
returned image URL to the console. The goal is to confirm the model works, evaluate
output quality, and measure latency. Test with at least 5 different user photo +
product image combinations and note which model parameters produce the best results.

# Out of scope for now
- Browser extension (deferred to Phase 2)
- Generation history or output gallery
- Share-to-social functionality
- Admin dashboard
- Multiple avatar profiles
- Subscription billing or auto-reload credits

# Rules
- Before making changes, inspect the existing codebase and summarize the current architecture.
- Do not build features outside this chunk.
- Do not build anything listed as out of scope.
- Do not refactor unrelated files.
- All Replicate API calls must go through a backend/server route — never expose the API key client-side.
- Do not store any user photos at this stage — use local test files only.
- After implementation: list which files were changed.
- After implementation: explain how to test this locally.
- Ask before adding new libraries or changing the stack.
