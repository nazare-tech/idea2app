# Plan: Build Your App Handoff (AI Prompts Section Redesign)

## Goal
Turn the derived AI Prompts workspace section into a non-technical-friendly "Build Your App" handoff. Users should immediately understand what the section is for, copy one prompt or all prompts with one click, follow a numbered prompt sequence generated by the First Version Plan, and copy their chosen mockup image so the recommended AI build tool can match the design. Reduce the friction between "Maker Compass recommended a tool" and "the user is actually building in that tool."

## Decisions Already Made (with the user)
1. **Multi-prompt UI**: Numbered stepper cards. Vertical "Prompt 1 of N" cards in build order, first card expanded, later cards collapsed (accordion behavior), each with its own copy button. Not tabs; tabs hide the sequence and imply the prompts are interchangeable.
2. **Prompt source**: Change First Version Plan generation. The FVP prompt contract will generate true paste-ready prompts, one per build-sequence step, instead of only the single "Next Prompt for AI Coding Tool."
3. **Section name**: Rename the user-facing section from "AI Prompts" to "Build Your App" (nav label, masthead, skeleton/empty states, explainable-term copy). Anchor IDs and the `ai-prompts` nav key stay stable so existing links and tests keep working.
4. **Mockup handoff**: Add a "Copy image" button next to the existing Download button on each mockup option, and embed the mockup thumbnail with copy/download actions inside the handoff stepper's final step.

## Assumptions
- The Recommended AI Build Tool card, its allowed-tool list, and its decision rules are unchanged; this work changes how prompts are packaged and presented, not which tool is recommended.
- Legacy First Version Plan documents (single "Next Prompt for AI Coding Tool" section) must keep rendering without regeneration; the stepper degrades to a single prompt card.
- "AI Build Guardrails" is not generated by the current FVP prompt contract (verified: no such heading in `src/lib/prompts/mvp-plan.ts`); it is a legacy-only section. The new contract folds guardrail rules into each generated prompt so every prompt is self-contained.
- `AiPromptsDocumentBlocks` currently receives `prdContent`, `mvpContent`, `projectId`; `scrollable-content.tsx` already has `mockupsData.content` in scope, so threading mockup content into the section is a prop change, not a data-loading change.
- Existing primitives are reused: `useCopyFeedback` + `copyTextToClipboard` (`src/lib/clipboard.ts`), the mockup image proxy (`/api/mockups/image`), and the mockup download handler in `src/components/ui/mockup-renderer.tsx`.
- No new database tables, queue items, or PDF export types; the section stays derived.

## Clarifying Questions
1. Should each generated build prompt be fully self-contained, or reference a shared context block pasted once?
   - Recommendation A: Self-contained. Each prompt carries a compact context header (product, target user, stack, out-of-scope) plus step-specific build/test instructions and rules.
   - Trade-off: Longer generated output, but prompts work when pasted into a fresh chat, which is what non-technical users actually do.
   - Recommendation B: Shared "project brief" prompt pasted once, then thin per-step prompts.
   - Trade-off: Shorter output, but breaks when the user starts a new chat or the tool loses context; harder to explain.
   - Selected: Recommendation A, with an explicit instruction to keep the shared header compact (under ~10 lines) so total output stays within budget.
2. How many prompts should the FVP generate?
   - Recommendation A: One prompt per build-sequence step, with the contract instructing 4-7 steps (the sequence already trends to this size).
   - Trade-off: Output length grows by roughly 300-500 tokens per prompt; `FIRST_VERSION_PLAN_MAX_TOKENS` (16,384 in `src/lib/first-version-plan-config.ts`) needs headroom review and likely a bump.
   - Recommendation B: Prompts for only the first 3 steps plus a "how to ask for the next step" template.
   - Trade-off: Cheaper, but reintroduces the friction this plan exists to remove.
   - Selected: Recommendation A, with a max-tokens bump if prompt-length estimates require it.
3. Should the new prompt-pack section replace §10 "Next Prompt for AI Coding Tool" or live alongside it?
   - Recommendation A: Replace. New contract heading "Build Prompts" with `### Prompt N: [Chunk name]` subsections, each containing one fenced text block. Parser keeps the old heading as a legacy alias.
   - Trade-off: Renderer and parser must handle both shapes, but the document stays non-redundant.
   - Recommendation B: Keep §10 and append a new section.
   - Trade-off: Duplicated prompt-1 content, larger output, confusing document.
   - Selected: Recommendation A.
4. Where should the redesigned section's components live?
   - Recommendation A: Extract the Build Your App blocks into a new `src/components/analysis/build-handoff-blocks.tsx` (or similar), leaving `planning-document-blocks.tsx` (3,500+ lines) to shrink rather than grow.
   - Trade-off: Small import churn, but the file is already oversized and this is a scoped architecture improvement consistent with project defaults.
   - Recommendation B: Keep everything in `planning-document-blocks.tsx`.
   - Trade-off: No churn, but the file keeps growing past maintainability.
   - Selected: Recommendation A.

## Recommended First Step
Add focused regression tests for the new prompt-pack parser (multi-prompt extraction, legacy single-prompt fallback, malformed content) before changing the FVP prompt contract, so the prompt/parser/renderer sync rule is enforced by tests from the start.

## Plan

### Phase 1: Frontend handoff shell (no AI contract change)
1. **Rename to "Build Your App"**: update the nav label in `src/lib/document-sections.ts`, the masthead in the AI Prompts blocks, skeleton/empty-state labels in `src/components/layout/scrollable-content.tsx`, and the explainable-term entry in `src/lib/explainable-terms.ts` (plain-English description of what the section is and how to use it). Keep the `ai-prompts` nav key and all `ai-prompts-*` anchor IDs stable. Update `document-sections.test.ts` label expectations.
2. **Plain-English intro + handoff stepper**: new component at the top of the section with three numbered steps that mirror the strategic "bias to the next action" principle:
   - Step 1: "Open [Tool name]" button using the URL already parsed by `getRecommendedTool()`.
   - Step 2: "Copy your first prompt" with a copy button wired to the first available prompt.
   - Step 3: "Add your design" with the embedded mockup thumbnail (see item 5) and copy/download actions, plus one sentence explaining the image is a design reference for the tool.
   The stepper copy contains no jargon, no em dashes, and follows the Pencil visual idioms (Fira Mono uppercase kickers, rounded dashboard surfaces, ease-out-expo entrance).
3. **Copy affordances on existing blocks**: add a copy button to the `FvpPromptBlock` terminal chrome bar using `useCopyFeedback`, and a "Copy all prompts" action that concatenates available prompts with numbered separators. Legacy guardrails sections get a copy button when present.
4. **Image clipboard helper**: extend `src/lib/clipboard.ts` with `copyImageToClipboard` using `ClipboardItem` with the promise-wrapped blob variant for Safari, and a graceful fallback that triggers the existing download path with feedback telling the user the image was downloaded instead. Unit-test the fallback ladder the same way `copyTextToClipboard` is tested.
5. **Mockup copy + embed**: add "Copy image" beside Download in `src/components/ui/mockup-renderer.tsx` (reusing the proxy-fetch logic from `handleDownload`), and pass `mockupContent` from `scrollable-content.tsx` into the Build Your App section so the stepper's Step 3 renders the first mockup option's storyboard thumbnail with copy/download. If no mockup exists yet, Step 3 renders a link back to the Design Mockups section instead.
6. **Component extraction**: move the Build Your App section components out of `planning-document-blocks.tsx` into a dedicated module as part of touching them.

### Phase 2: FVP prompt contract change (multi-prompt generation)
7. **Contract change in `src/lib/prompts/mvp-plan.ts`**: replace §10 "Next Prompt for AI Coding Tool" with "Build Prompts": one `### Prompt N: [Chunk name]` subsection per build-sequence step (4-7 steps), each containing exactly one fenced `text` block that is fully paste-ready and self-contained: compact shared header, what earlier prompts built, build-only-this-chunk instruction, the step's "test before moving on" criteria, and the rules block (current Next Prompt rules, kept short). Update the Output Prioritization and Quality Bar sections to reference the prompt pack.
8. **Token budget**: estimate worst-case prompt-pack length; bump `FIRST_VERSION_PLAN_MAX_TOKENS` in `src/lib/first-version-plan-config.ts` if needed (likely 16,384 → ~20,480) and confirm the shared 480s planning-document timeout still has margin. Prompt Lab inherits both automatically through the shared request builder.
9. **Parser**: extend the FVP/AI-Prompts extraction (in `src/lib/mvp-plan-document.ts` and the section-alias logic used by the Build Your App blocks) to return a typed array of prompts `{ index, title, body }` from "Build Prompts", with the legacy "Next Prompt for AI Coding Tool" heading as a single-prompt fallback alias. Malformed prompt subsections fall back to rendering the raw section content, never dropping it.
10. **Stepper card renderer**: render the prompt array as numbered stepper cards ("Prompt 1 of N", chunk name, one-line plain-English summary of what the step builds, terminal-style body, per-card copy button, copied feedback). First card expanded, later cards collapsed. Legacy documents render one card and no count. "Copy all prompts" concatenates the pack.
11. **Tests kept in sync (project rule: prompt, parser, renderer, navigation, tests move together)**: update `src/lib/planning-prompts.test.ts` (contract text), `src/lib/mvp-plan-document.test.ts` (multi-prompt + legacy parsing), `src/components/analysis/planning-document-blocks.test.tsx` plus new tests for the extracted module (stepper rendering, copy wiring, legacy fallback), and `src/lib/document-sections.test.ts` (labels).
12. **Docs**: update `PROJECT_CONTEXT.md` (AI Prompts section description, FVP contract description, renamed nav label, new clipboard helper) and `docs/plans/backend-change-history.md` for the contract/token changes.

## Architecture Improvement Opportunities
- **Contract-sync discipline**: Selected. The prompt, parser, renderer, nav, and test changes ship in one phase with parser regression tests written first, per the cross-cutting defaults in `PROJECT_CONTEXT.md`.
- **Legacy fallback safety**: Selected. Old FVP documents keep rendering through the single-prompt alias; malformed new documents fall back to raw section content instead of dropping it.
- **Shared clipboard module**: Selected. Image copy lands in `src/lib/clipboard.ts` beside text copy so mockup renderer, stepper, and any future surface share one fallback ladder.
- **Oversized component module**: Selected. Extracting the Build Your App blocks starts shrinking `planning-document-blocks.tsx` instead of growing it.
- **Prompt Lab parity**: Selected implicitly. Because production and Prompt Lab share the FVP request builder, the new contract is immediately testable in Prompt Lab without extra wiring.

## Milestones
- Phase 1 shipped: section renamed, intro + stepper live, Next Prompt and guardrails copyable, mockup copy-image and embedded thumbnail working, clipboard helper tested.
- Phase 2 contract shipped: FVP generates the Build Prompts pack, parser extracts it with legacy fallback, stepper renders numbered prompt cards with per-card and copy-all actions.
- Verification complete: focused tests, typecheck, and lint pass; a fixture or Prompt Lab run shows the new document shape rendering end to end; `PROJECT_CONTEXT.md` and backend change history updated.
