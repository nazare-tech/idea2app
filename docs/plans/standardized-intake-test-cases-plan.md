# Plan: Standardized Intake Test Cases

## Goal
Create a durable, reusable set of three business-idea fixtures for the Maker Compass intake wizard, including default answers to common follow-up question categories. This plan originally referred to Idea 1 as the default; the follow-up live capture split the fixture into variants and made Idea 1.1 the default.

## Assumptions
- The canonical user flow is `/projects/new`.
- Step 2 questions are generated by the OpenRouter-backed intake question API and may vary over time.
- The required `primary-platform` question is stable and must be answered with one canonical option ID.
- This change should be documentation-only unless a future task asks for executable tests.
- Live UI/API question generation is useful but not required for this task because it may require credentials and paid AI calls.

## Clarifying Questions
1. Where should the reusable test cases live?
   - Recommendation A: Add a dedicated markdown file under `docs/guides/` and reference it from `AGENTS.md`.
   - Trade-off: Keeps the full fixture easy to read and update, while adding one lightweight pointer to agent instructions.
   - Recommendation B: Put the full fixture directly in `AGENTS.md`.
   - Trade-off: Makes it impossible to miss, but bloats the instruction file with operational details.
   - Selected: Recommendation A, because this is a living fixture that future agents should update.
2. How should changing AI-generated questions be handled?
   - Recommendation A: Record canonical answers by question category, plus an append-only observed-question log for future runs.
   - Trade-off: Stable enough to use now while still adapting when the model asks new question text.
   - Recommendation B: Record only exact question text from one local run.
   - Trade-off: More literal, but brittle because the UI intentionally generates questions dynamically.
   - Selected: Recommendation A, because the app does not guarantee identical questions.
3. Should I run the full UI creation flow now?
   - Recommendation A: Do not create a live project or invoke AI generation just for this fixture; inspect the code and document safe manual update instructions.
   - Trade-off: Avoids spending API money or touching user data, but does not capture a fresh live question set today.
   - Recommendation B: Run the authenticated UI and generate a project from each idea.
   - Trade-off: Captures exact current questions, but may spend external API budget and create durable project data.
   - Selected initially: Recommendation A, due to the repo safety rule around money and production-like data changes.
   - Correction: The user clarified on 2026-06-30 that this was the wrong default for future Maker Compass QA. Small expected AI/API spend for real local verification is acceptable when it avoids more future work and produces durable test evidence. `docs/plans/recommendation-selection-rules.md` now records this as a learned rule.

## Risks And Mitigations
- Risk: Future generated questions do not match the recorded examples.
  - Mitigation: Include category-based answer rules and an append-only log for newly observed questions.
- Risk: Agents accidentally use Idea 2 or Idea 3 as the default.
  - Mitigation: State the default in both `AGENTS.md` and the fixture.
- Risk: Live UI verification may spend AI/API budget.
  - Mitigation: Treat bounded local QA spend as acceptable when it captures durable evidence; stop only for open-ended spend, production-impacting spend, missing credentials, destructive data changes, or explicit safety blockers.

## Rollback Or Recovery
Revert the added docs/testing fixture and remove the `AGENTS.md` pointer. No runtime behavior, database schema, or dependency changes are involved.

## Critique

### Software Architect
- A documentation fixture is appropriate because the intake questions are dynamic and no executable test harness was requested. A future enhancement could convert the fixture into a JSON/YAML source if automation needs it.

### Product Manager
- The fixture creates repeatable input context for report-quality comparisons while preserving two alternate ideas for targeted checks.

### Customer Or End User
- Stable examples reduce inconsistent testing and make generated report regressions easier to spot.

### Engineering Implementer
- The fixture should avoid overly specific exact-question assumptions. The most durable unit is the answer intent per question category and the stable platform option ID.

### Risk, Security, Or Operations
- This documentation change avoids credentials, secrets, paid API usage, project creation, and user data mutation.
