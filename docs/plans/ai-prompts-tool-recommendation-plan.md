# Plan: AI Prompts Tool Recommendation

## Goal
Narrow the AI coding-tool research set and add a single primary recommended tool to the top of the AI Prompts handoff section. The recommendation should be generated as part of the First Version Plan / AI Prompts content and rendered using the external-link visual language used for competitor entries.

## Assumptions
- Remove Sourcegraph, JetBrains, Tabnine, Zed, and Windsurf naming from the research database because they are not product-facing handoff recommendations for this flow.
- Keep Devin, but merge Devin Desktop into the Devin profile instead of treating it as a separate tool.
- Keep one primary recommended tool only. No ranked list in the UI.
- The visible recommendation should come from generated planning content so different project variants can produce different recommendations.
- For old projects without the new section, the AI Prompts section should continue rendering without a recommendation card.

## Clarifying Questions
1. Should recommendation selection be deterministic code or generated by the planning prompt?
   - Recommendation A: Add a required First Version Plan section and renderer support. This keeps the recommendation explainable in the generated artifact and lets Idea 1.1 vs 1.2 differ naturally.
   - Trade-off: The LLM can occasionally choose unexpectedly, so the prompt must constrain available tools and decision rules.
   - Recommendation B: Hardcode a local rule engine in the renderer. This is predictable but may ignore nuanced project context in the generated plan.
   - Trade-off: It can become stale and opaque as tool capabilities change.
   - Selected: Recommendation A, because the user wants this tied to generated project handoff output.
2. Should unavailable/removed tools remain in the markdown as "not recommended" history?
   - Recommendation A: Remove them from active research profiles and summary tables, with a short maintenance note only if useful.
   - Trade-off: Loses old comparison detail, but matches current product direction.
   - Recommendation B: Keep them under an archived/deprioritized section.
   - Trade-off: More clutter and easier for future UI to accidentally surface them.
   - Selected: Recommendation A, because the user explicitly asked to remove them.
3. How should we verify with Idea 1.1 and 1.2?
   - Recommendation A: Use the real local UI flow with configured e2e credentials and capture evidence, creating two projects if local auth/API dependencies allow it.
   - Trade-off: May spend local AI/API credits and can be blocked by allowance/auth/API availability.
   - Recommendation B: Use unit tests and prompt snapshots only.
   - Trade-off: Faster, but does not prove the user-visible generated recommendation.
   - Selected: Recommendation A, per repo instruction and the user request to create projects for comparison.

## Risks And Mitigations
- Risk: Generated tool recommendation varies unpredictably.
  - Mitigation: Constrain allowed tools and provide selection rules by platform/backend/security need.
- Risk: Existing projects lack the new section.
  - Mitigation: Render the card only when the section exists.
- Risk: Creating two real projects consumes local allowance or AI spend.
  - Mitigation: Use the configured local QA flow; stop if auth, allowance, credentials, or provider availability blocks the real UI path.

## Rollback Or Recovery
Revert the prompt and renderer changes. Existing generated documents are markdown and remain readable; new recommendation sections would simply stop rendering as a special card.

## Critique

### Software Architect
- Generated markdown plus renderer parsing matches the existing AI Prompts architecture and avoids adding a new database field.

### Product Manager
- One recommendation is clearer than a tool directory at the handoff moment. The full research database remains useful for future logic.

### Customer Or End User
- Users should see a concrete next step, link, expected cost, and why that tool fits their project.

### Engineering Implementer
- The highest-risk work is parsing loosely generated markdown. Keep the expected format simple and add tolerant aliases.

### Risk, Security, Or Operations
- The recommendation must not push private/backend-heavy projects into cloud app builders without warning. Include privacy/backend selection rules in the prompt.
