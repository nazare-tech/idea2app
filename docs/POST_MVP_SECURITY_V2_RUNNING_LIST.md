## Running List

### 1. Replace or isolate live HTML mockup previews

Status: deferred until after MVP launch
Source finding: `docs/SECURITY_REVIEW_2026-04-25.md`, finding 1

Current MVP decision:
- Keep live HTML previews enabled for now.

Preferred future direction:
- Deprecate live HTML previews.
- Replace them with static image previews generated by a current image model.

Alternative isolation options:
- Serve previews from a separate isolated preview origin.
- Remove `allow-same-origin` from preview iframes.
- Remove `allow-scripts` unless scripts are truly required.
- Add strict CSP sandboxing for preview responses.
- Require auth and verify each preview URL belongs to a saved mockup for a project owned by the current user.

Review questions:
- Are users still relying on interactive HTML previews?
- Is image-only preview quality good enough to replace HTML?
- Can preview rendering move to a separate origin without slowing the MVP roadmap?

### 2. Move from simple server-side rate limiting to distributed rate limiting

Status: deferred until traffic/deployment needs justify it
Source finding: `docs/SECURITY_REVIEW_2026-04-25.md`, finding 8

Current MVP decision:
- Start with a simple server-side limiter.

Future options:
- Use Upstash Redis or another shared store for Vercel/serverless-safe limits.
- Add per-user and per-IP limits.
- Add stricter limits for PDF export, AI generation, Stitch generation, auth handoff, waitlist, and checkout/session creation.

Review questions:
- Is the app running on multiple serverless instances or regions?
- Are there signs of abuse or accidental high-volume traffic?
- Which endpoints are generating the most spend or CPU load?

### 3. Dependency security upgrade pass

Status: deferred to a separate pass
Source finding: `docs/SECURITY_REVIEW_2026-04-25.md`, finding 13

Current MVP decision:
- Defer dependency upgrades until after behavioral security fixes.

Future work:
- Update direct dependencies first: `next`, `jspdf`, `mermaid`, `puppeteer`, and `@google/stitch-sdk` when patched.
- Re-run `npm audit --omit=dev`.
- Regression-test PDF export, Mermaid rendering, mockup generation, and app build.

Review questions:
- Are patched versions available without breaking the app?
- Are any advisories actively exploitable in production paths?
- Can upgrades be grouped safely, or should they be split into smaller PRs?

### 4. Long-term preview architecture decision

Status: open design decision
Source finding: `docs/SECURITY_REVIEW_2026-04-25.md`, findings 1, 4, and 12

Options:
- Static image previews only.
- Static screenshots generated server-side from isolated rendering.
- Dedicated preview service/origin.
- Keep HTML downloads but remove embedded live preview.

Review questions:
- Which preview format best supports the user workflow?
- Can generated preview artifacts be treated as untrusted files end to end?
- Is there still a need for downloadable HTML once image previews exist?

### 5. Move intake project creation into one database transaction

Status: deferred beyond MVP hardening
Source finding: `docs/SECURITY_REVIEW_2026-04-25.md`, finding 6

Current MVP decision:
- Use a short-lived service-role `project_creation_locks` row plus a second allowance check immediately before project insertion.

Future work:
- Replace the app-level lock with a single Supabase RPC or transaction that checks allowance and inserts `projects`, `project_intakes`, `generation_queues`, and `generation_queue_items` under one database boundary.
- Use an advisory lock or serializable transaction inside that RPC.

Review questions:
- Are users hitting project creation conflicts or lock timeouts?
- Can project/intake/queue creation move fully into SQL without making the route hard to maintain?
