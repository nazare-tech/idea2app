# Plan: Idea Intake Wizard

## Goal
Implement the new Figma idea intake wizard as the canonical project-start flow. A user should see this wizard whenever they click a new project action, and users who type an idea on the landing page before signing in or signing up should keep that idea through auth and land in the wizard with the idea already populated. The wizard replaces the current initial Prompt Chat Q&A, generates AI-driven answer-chip questions in Step 2, creates the project only after Step 2, stores the finished structured intake in a new `project_intakes` table, stores a human-readable summary in `projects.description`, then hands off to the existing workspace for refinement and downstream document generation.

## Assumptions
- The provided Figma node `65:1658` is the intended MVP: Step 1 "Idea Brief" and Step 2 "Tell us a bit more".
- Step 1 has two valid states: empty with example idea cards, and prefilled with the landing-page idea or selected example idea.
- Step 2 should use AI-generated structured questions, not the fixed example questions from the design.
- The current `projects.description` field is shown on dashboard cards via `DashboardProjectCard`, so it should remain human-readable and should not store raw JSON.
- Structured intake data should live in a new `project_intakes` table so it can support analytics, querying, regeneration, and future personalization.
- AI pipelines should receive formatted intake context from `project_intakes` plus the human-readable project summary, rather than relying only on `projects.description`.
- The current Prompt tab initial Q&A should be deprecated or disabled for newly created intake-wizard projects. Prompt Chat can remain available after creation for follow-up refinement, but it should not repeat the wizard's intake questions.
- A short-lived server-side pending-intake record is acceptable for auth handoff, because only an opaque token goes in the URL and the actual idea stays out of browser history/referrers.
- The landing page does not currently have the described idea entry box, so that will be a new component rather than a simple wiring change.
- Auth still uses Supabase email/password and Google OAuth through the shared `AuthFormContent`.
- Project name generation currently happens inside `/api/prompt-chat`; because the wizard replaces initial Prompt Chat Q&A, name generation must move into the intake completion path or a shared helper.

## Decisions
1. The wizard creates the project only after Step 2 is completed.
2. Step 2 answers are stored in a new `project_intakes` table as versioned JSON for the first implementation.
3. Step 2 questions are generated by AI from the Step 1 idea.
4. The AI question schema must support single-select chips, multi-select chips, optional free-text "Other" answers, and potentially plain text answers.
5. The wizard replaces the initial Prompt Chat Q&A entirely.
6. Pending landing-page idea text should also be stored in a short-lived server-side pending-intake record when auth/email confirmation durability is needed.
7. The URL should carry only an opaque `intake` token, never the raw idea text.
8. Example ideas should come from a configurable list in code, seeded with the three examples from the Figma design.
9. The wizard should be a reusable full-page route used by both landing and dashboard entry points.
10. Back from Step 2 should return to Step 1 while preserving Step 2 selections.
11. If the user changes the Step 1 idea after Step 2 questions have been generated, the Step 2 questions and answers should be invalidated and regenerated.
12. Logged-in users should normally not see the landing page, but if they do, entering an idea and clicking the CTA should skip auth and go straight to the wizard.
13. The final Step 2 button should say "Create project".
14. Step 2 is required for now.
15. AI Step 2 question generation should not consume credits or project quota; it is a free onboarding operation.
16. The billing direction is shifting away from credits toward plan-based project allowances, so the intake wizard should avoid introducing new credit dependencies.
17. Step 2 should generate 4-5 questions to match the current Prompt Chat intake depth.
18. Project name generation should happen after Step 2 if the current AI name-generation path can be reused. If not, create the project as "Untitled" and allow the user to edit it later.
19. The landing-page idea box should be visible on the landing page, but its visual design is not included yet and should be implemented after design input is available.
20. Anonymous pending-intake records should expire after 24 hours.
21. Monthly project allowance must be enforced during project creation. If the user is at their plan limit, do not create a new project.
22. The existing Prompt Chat UI should be removed from the primary new-project flow, but the code can be retained or moved to a non-primary/internal reusable location for future reuse.
23. Old Prompt Chat conversation data can be migrated into `project_intakes` where straightforward. If the old data is incomplete, redundant, or blocks the new flow, deleting/archiving it is acceptable.

## Recommended First Step
Refactor the existing Prompt Chat question-generation and project-name logic into reusable server-side intake services before building UI. The question service should accept the Step 1 idea and return a typed JSON question set with answer chips, selection rules, and optional free-text fields. The name service should accept the completed intake JSON and return a short title when AI generation is available. This validates the hardest behavior first and lets the wizard UI remain a clean renderer for stable server contracts.

## Parallelization Order
1. Contract setup first: Workstreams A and B can run in parallel and should establish the schema/types used by everyone else.
2. API/backend next: Workstreams C and D can run after A/B contracts are stable.
3. UI in parallel with backend: Workstreams E and F can start with mocked API contracts, then integrate against C/D.
4. Workspace cleanup after contracts: Workstream G can begin after A/B and should integrate once F can create a new intake-backed project.
5. Legacy migration after schema: Workstream H can start after A and should finish after G clarifies how old Prompt Chat data is no longer read.
6. Final integration: Workstream I owns merge validation, documentation, and cross-flow QA.

## Risks And Mitigations
- Risk: Putting raw idea text in the URL would leak sensitive idea content through browser history, logs, referrers, and analytics. Mitigation: put only an opaque pending-intake token in the URL and store the idea server-side until claimed or expired.
- Risk: AI may return malformed or awkward chip questions. Mitigation: enforce a strict JSON schema, validate/repair server-side where reasonable, and fall back to a curated default question set if generation fails.
- Risk: Step 2 replaces Prompt Chat Q&A, so project naming and summarization behavior currently tied to Prompt Chat could regress. Mitigation: extract shared name/summary helpers and call them during intake completion.
- Risk: Introducing `project_intakes` adds schema/RLS work and migration complexity. Mitigation: keep `projects.description` as the readable summary for existing UI, add focused RLS tests/manual checks, and keep the intake JSON schema versioned.
- Risk: Deleting old Prompt Chat data could remove useful context from existing projects. Mitigation: migrate useful summaries first where straightforward; only delete/archive rows that are redundant, incomplete, or incompatible with the new flow.
- Risk: The move from credits to project allowances could create inconsistent enforcement if mixed into this feature halfway. Mitigation: make intake AI free and enforce project quota at project creation before any project/intake rows are inserted.
- Risk: OAuth redirects can create open redirect vulnerabilities if `next` is not constrained. Mitigation: allow only relative internal paths and preferably only `/projects/new`, `/dashboard`, and `/projects/*`.
- Risk: Replacing `/projects/new` could break any code expecting immediate project creation. Mitigation: search all callers and keep `/projects/new` as the stable URL while changing only its behavior.
- Risk: The Figma uses fixed desktop dimensions that may not fit mobile. Mitigation: implement the visual language responsively, not as fixed pixel-perfect absolute positioning.

## Critique

### Software Architect
- The plan preserves the existing workspace and downstream document architecture, but it now moves the initial intake contract out of Prompt Chat. The right technical move is to extract shared prompt/name helpers instead of copying logic between `/api/prompt-chat` and the wizard APIs. Moving structured intake into `project_intakes` is cleaner than overloading `projects.description`, because dashboard cards and other UI already display `projects.description` as plain text.

### Product Manager
- The plan now has a clearer onboarding story: landing idea, AI-guided intake, then workspace. The product risk is that AI-generated chips may feel random unless the questions are concise and visibly relevant. Making Step 2 required increases context quality but adds friction, so completion rates should be watched after launch.

### Customer Or End User
- The design gives users a more guided start and reassures them that their typed idea was remembered. A URL token backed by server-side pending storage is a better experience than `sessionStorage` alone because it can survive email confirmation in a new tab without exposing the idea in the URL.

### Engineering Implementer
- The safest implementation is to keep `/projects/new` as the single entry point and move creation to final submit. The highest-risk details are typed AI output parsing, pending-token lifecycle, `project_intakes` RLS, project allowance checks, auth redirect plumbing, Prompt Chat removal from the primary flow, and making the wizard responsive without copying Figma's generated absolute layout.

### Risk, Security, Or Operations
- The biggest security issues are redirect handling, idea leakage, and RLS on the new intake tables. Redirects need an explicit internal-path allowlist, and the URL should carry only opaque tokens. Pending-intake records need a 24-hour expiration and claim semantics so abandoned ideas do not live forever and tokens cannot be abused. Project allowance checks must be transactional enough to avoid partial project creation.
